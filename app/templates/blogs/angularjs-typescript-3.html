Lets Recap, Why Angular?

Angular is MVC on the client. Define your Model, View, and Controller then let angular do the heavy lifting. Service Injection, DOM binding, Directives... Angular brings alot to the table.

Why use Directives?

First of all, lets define directive again.

Directive - Sometimes logic is used multiple places in an app, and it seems it can be boiled down to a certain shared component. This is where a Directive steps in. A Directive is almost like a mini MVC app. It has a Controller and a View. Directives can be defined in HTML as it's own element type.

At my job, I write enterprise-level software. When you are writing software to do some enterprise level functions, odds are the component you need is already been written and tested. So rather than re-invent the wheel each time, we should share the code between modules. This way, if you need to make a change across the entire module, you only need to update one file.
One of my solutions I work on has a SharedUI project that contains many directives. We recently decided to change the style of all the text fields across all projects in the solution. It was an easy change because I went into the directive's code, added a class and I was done.
